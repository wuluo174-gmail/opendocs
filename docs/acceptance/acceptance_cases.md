> 本文件只定义最终通过/失败判定、环境与语料契约、`TC-001 ~ TC-021` 可执行验收用例，以及 `run_acceptance.py` 的预期行为。
> 本文件不定义阶段顺序、阶段 `status` 或产品架构。

- 规则裁决顺序与文件职责矩阵以 `docs/specs/OpenDocs_工程实施主规范_v1.0.md` 为准，本文件不重复抄写整段裁决链。

## 1. 验收目标

验收要证明的不是“系统看起来能跑”，而是以下红线真实成立：

1. 证据优先：事实性回答默认带引用，且引用可打开、可核验。
2. 本地优先：`Local-Only` 模式外网请求为 0。
3. 语义索引优先：索引、检索和分类真正使用内容语义；统计 fallback 只能作为显式降级模式。
4. 内容优先分类：命名混乱、跨项目混放时，分类与归档仍以内容证据为主。
5. 写操作受控：`move / rename / create / save_output` 必须预览、确认、审计、可回滚。
6. 记忆不越权：记忆不能覆盖文档证据；冲突时以文档证据为准。
7. 可审计可恢复：关键链路可查询，最近一次批量写操作可回滚。
8. 可交付：一键验收、性能结果、静态检查、覆盖率、打包与用户文档齐备。

补充要求：

- 验收对象是规范符合性，不是旧实现延续性。
- 若现有实现与当前规范冲突，应判为待重构或失败，而不是把旧行为视为通过。

## 2. 整体验收判定

### 2.1 通过条件

以下条件同时满足，整体验收才算通过：

- `TC-001 ~ TC-021` 全部通过
- `TC-020` 覆盖的 `NFR-001 ~ NFR-023` 均有可核验证据
- 五项核心红线全部通过：
  - `Local-Only` 外发请求为 0
  - 语义模式可见，统计 fallback 仅能显式启用
  - 分类与归档在命名混乱、跨项目混放语料上仍以内容证据为主或进入待确认
  - 文件系统与用户输出写操作具备预览、确认、审计、回滚
  - 事实问答默认附引用，证据不足时拒答
- 不接受以旧版实现行为为理由降低用例断言、跳过必需产物或放宽失败判定
- 无 Blocker 级缺陷
- 自动化报告与人工抽查结论一致

### 2.2 失败分级

- Blocker：破坏文件安全、证据可信性、隐私红线、回滚能力或主流程不可用
- Major：主要功能不完整、核心用例失败、性能严重不达标
- Minor：边界显示问题、文案问题、个别非核心平台差异

### 2.3 执行分层

验收至少分三层执行：

1. 自动化层：`scripts/run_acceptance.py`
2. 人工冒烟层：UI、引用面板、预览确认、风险提示、记忆管理页
3. 性能与交付层：性能阈值、打包、文档与跨平台回归

## 3. 环境、语料与报告契约

### 3.1 最低环境

- Python 3.11
- 支持 SQLite FTS5
- 可用的本地文件系统沙箱目录
- 如需覆盖云端路径，必须至少有一个可配置 Provider；若无密钥，使用 mock provider 或其他本地无外发测试替身
- 报告建议使用 `zh-CN`

### 3.2 测试目录建议

```text
.tmp/
├─ appdata/
├─ corpus/
├─ output/
├─ reports/
└─ sandbox_fs/
```

### 3.3 语料契约

验收语料生成入口以 `docs/acceptance/tasks.yaml` 中 `default_commands.fixture_generation` 为准。

若需要自定义输出目录，脚本应支持如下参数形态：

```bash
scripts/generate_fixture_corpus.py --profile acceptance --output .tmp/corpus
```

生成结果至少包含：

- 项目周报 80 篇
- 会议纪要 80 篇
- 月报/季报 40 篇
- 研究笔记 40 篇
- 合同/报价/交付清单 20 篇
- 冲突版本文档 20 组
- 破损/空白/极短文档 20 个
- 黄金问题集 >= 100 条
- 同一语义不同表述或项目别称样本 >= 30 组
- 中性命名或误导性命名文档 >= 20 个
- 归档规划场景样本 >= 20 组
- 上述归档规划样本中，至少 10 组同时满足“命名混乱或中性命名 + 跨项目混放 + 仅靠路径/时间无法判定”

### 3.4 报告输出契约

`run_acceptance.py` 至少输出：

- `report.json`
- `report.md`

建议 JSON 结构：

```json

  "started_at": "2026-04-08T00:00:00Z",
  "finished_at": "2026-04-08T00:20:00Z",
  "summary": {
    "passed": 21,
    "failed": 0,
    "blocked": 0,
    "risk_count": 0
  },
  "cases": [
    {
      "id": "TC-001",
      "status": "passed",
      "duration_sec": 4.2,
      "severity": "Blocker",
      "artifacts": ["logs/audit.jsonl", "reports/tc-001.json"]

  "metrics": {
    "search_p95_sec": 1.7,
    "citation_open_p95_sec": 0.8,
    "cold_start_sec": 5.6
  }

```

Markdown 报告至少包含：

- 执行环境
- 版本信息
- 用例结果
- 风险项
- 未完成项
- 关键性能指标
- 最终结论

## 4. 共享验收契约

### 4.1 TaskEvent 契约

凡用例中出现“写入结构化 `TaskEvent`”，均指满足主规范中的 `TaskEvent` 最小字段契约，而不是各处自定义一份缩水字段。

### 4.2 SourceRoot / ScanRun 契约

凡用例中出现“根目录配置已保存”或“扫描记录已生成”，默认同时检查：

- `SourceRoot` 是真实持久化对象，而不是脚本临时参数
- `Document.source_root_id` 与 `ScanRun.source_root_id` 可回查到真实 `SourceRoot`
- `ScanRun` 至少包含 `source_root_id`、统计字段、`trace_id`、开始/结束时间与状态

### 4.3 IndexArtifact 契约

凡用例中出现“dense 索引可用”“语义模式可见”或“降级状态可见”，默认同时检查：

- `IndexArtifact` 是派生工件状态对象，不替代源事实
- 至少可区分 `ready / stale / building / failed`
- 至少可读取 `artifact_path`、`embedder_model`、`embedder_signature`

### 4.4 FileOperationPlan 预览契约

凡用例中出现“归档预览”或“执行前预览”，默认同时检查 `FileOperationPlan.preview_json` 至少包含：

- 逐项变更清单
- 原路径与目标路径
- 内容依据摘要
- 风险提示
- 最近一次批量回滚所需最小信息

### 4.5 语义模式契约

凡用例中出现“语义模式可见”，均指系统能够明确区分：

- 正常语义模式
- 统计 fallback 降级模式

且两者都能在 UI、状态面板、日志或审计中被确认。

### 4.6 内容证据分类契约

凡用例中出现“分类依据”，均指至少包含内容证据；文件名、路径、时间只能作为辅助弱信号。

### 4.7 记忆契约

凡用例中出现记忆断言，默认同时检查：

- `M0` 不落盘
- 持久化 `MemoryItem.scope_type` 不允许 `session`
- `candidate / disabled / expired / superseded` 记忆不参与正常召回
- 记忆冲突时以文档证据为准

## 5. 详细验收用例

### TC-001 新增根目录并扫描

- 级别：Blocker
- 覆盖：`FR-001`
- 类型：Auto
- 前置条件：
  - `.tmp/corpus/main` 已存在
  - 应用尚未配置该根目录
- 步骤：
  1. 调用 `SourceService.add_source(...)`
  2. 调用 `SourceService.scan_source(...)`
  3. 读取 scan 结果与审计日志
- 预期结果：
  - 根目录配置可保存并再次加载
  - `SourceRoot` 具备稳定显示根名、排除规则与配置版本等最小字段
  - 输出纳入、排除、失败统计
  - 生成可回查到 `source_root_id` 与 `trace_id` 的扫描记录
  - 生成审计记录
- 自动化断言：
  - 断言 source_root 持久化存在
  - 断言最新 `scan_run` 挂在真实 `source_root_id` 上
  - 断言扫描统计与语料实际数量一致
  - 断言审计中存在 `scan_source` 事件
- 需保留证据：
  - source 配置快照
  - 扫描结果
  - `scan_run` 快照
  - 审计日志片段
- 失败判定：
  - 不能保存根目录
  - 扫描统计错误
  - 无审计记录

### TC-002 不支持文件与单文件失败不拖垮批任务

- 级别：Blocker
- 覆盖：`FR-001`、`FR-002`
- 类型：Auto
- 前置条件：语料中存在不支持格式、损坏文档和无权限样本
- 步骤：
  1. 发起扫描与索引
  2. 记录失败文件列表
  3. 检查其余有效文件是否完成入库
- 预期结果：
  - 不支持文件进入排除清单
  - 失败文件被标记为 `failed` 或 `partial`
  - 其余文件照常完成索引
- 自动化断言：
  - 总任务未中断
  - 失败数 > 0 且成功数 > 0
  - Document 表中存在成功、失败两类状态
- 需保留证据：
  - 失败清单
  - 总任务结果
- 失败判定：
  - 任一单文件异常导致批任务整体失败
  - 失败未被记录

### TC-003 新增文件后的增量索引

- 级别：Blocker
- 覆盖：`FR-002`
- 类型：Auto
- 前置条件：已完成首次全量索引
- 输入特点：新增合法 `.md` 或 `.docx` 文档，文件名不含项目名或核心关键词
- 步骤：
  1. 写入新文件
  2. 触发 watcher 或主动增量索引
  3. 用自然语言或同义表达搜索该文档主题
- 预期结果：
  - 新文档写入 Document、Chunk、FTS、HNSW
  - `IndexArtifact` 或等价状态查询能表明 dense 派生工件仍处于可用状态
  - 文档可通过语义查询命中
  - 产生增量索引审计
- 自动化断言：
  - `indexed_at` 晚于新增时间
  - 新增 chunk 具备可追踪语义表示或显式降级标记
  - `IndexArtifact` 状态可读，且 embedder 信息未失联
  - 搜索结果包含新文档路径
- 需保留证据：
  - 新文档样本
  - 索引工件状态快照
  - 搜索命中结果
- 失败判定：
  - 新文件无法被搜索到
  - 监听延迟明显且无补偿机制

### TC-004 删除文件后的索引同步

- 级别：Blocker
- 覆盖：`FR-002`
- 类型：Auto
- 前置条件：存在可检索文档
- 步骤：
  1. 先搜索唯一关键词，确认命中
  2. 删除文件
  3. 触发 watcher 或手动增量更新
  4. 再次搜索同一关键词
- 预期结果：
  - Document 被标记为 `is_deleted_from_fs = true` 或从活跃结果移除
  - 检索结果中不再出现该文档
- 自动化断言：
  - 搜索结果为 0
  - 审计中存在 delete-from-fs 同步事件
- 需保留证据：
  - 删除前后搜索日志或截图
- 失败判定：
  - 已删除文件仍在结果中可见

### TC-005 混合搜索返回可引用结果

- 级别：Blocker
- 覆盖：`FR-003`、`FR-015`
- 类型：Hybrid
- 前置条件：
  - 完成全量索引
  - 黄金查询集已生成
- 输入要求：
  - 5 条定位型查询
  - 5 条同义表达查询
  - 至少 3 条查询的目标文档文件名不包含查询核心词
  - 3 组过滤条件组合
- 步骤：
  1. 使用 `SearchService.search(...)`
  2. 检查 Top10 结果
  3. 人工点击至少 2 个引用跳转
- 预期结果：
  - 每个结果含标题、路径、摘要、时间、分数、引用定位
  - 同义表达查询可依靠内容语义召回目标文档
  - 引用能打开文件或定位到片段附近
- 自动化断言：
  - 统计 Top10 命中率
  - 统计同义表达查询 Top10 命中率
  - 断言结果引用字段非空
- 需保留证据：
  - 查询与结果日志
  - 引用点击截图或录像
- 失败判定：
  - 返回结果无引用
  - 过滤器失效
  - 同义表达查询只能命中字面重合文档
  - 无法打开源文件或目录

### TC-006 事实问答默认附引用

- 级别：Blocker
- 覆盖：`FR-004`
- 类型：Auto
- 前置条件：至少一组问题具有确定答案且证据明确
- 步骤：
  1. 调用 `QAService.answer(...)`
  2. 检查答案结构、引用列表、`trace_id`
- 预期结果：
  - 所有事实性结论默认带引用
  - 引用指向真实存在的 chunk
  - 回答结构符合主规范要求
- 自动化断言：
  - `answer.citations` 非空
  - 关键事实被引用校验器接受
- 需保留证据：
  - 回答 JSON
  - 引用校验结果
- 失败判定：
  - 有事实结论但无引用
  - 引用无法解析到 chunk

### TC-007 证据不足拒答

- 级别：Blocker
- 覆盖：`FR-004`
- 类型：Auto
- 前置条件：黄金问题集中存在无解或证据不足问题
- 步骤：
  1. 调用 `QAService.answer(...)`
  2. 检查是否输出拒答结果
- 预期结果：
  - 返回“不足以可靠回答”或等价拒答
  - 给出已检查来源与建议下一步
  - 不伪造具体事实
- 自动化断言：
  - 结果类型为 `insufficient_evidence`
  - 不存在未被证据支撑的具体结论
- 需保留证据：
  - 拒答结果
  - 检索候选摘要
- 失败判定：
  - 明显胡答
  - 以记忆或猜测替代证据

### TC-008 冲突证据展示

- 级别：Blocker
- 覆盖：`FR-004`、`FR-009`
- 类型：Auto
- 前置条件：语料中至少存在 2 个版本冲突问题
- 步骤：
  1. 执行问答
  2. 检查系统是否识别冲突
  3. 读取冲突来源列表
- 预期结果：
  - 输出冲突说明而非单一确定结论
  - 至少展示 2 个冲突来源
  - 若有记忆冲突，标明记忆可能陈旧或错误
- 自动化断言：
  - `result_type == conflict`
  - `conflict_sources >= 2`
- 需保留证据：
  - 冲突回答
  - 来源 A/B 摘要
- 失败判定：
  - 冲突场景被硬性归并成单一结论
  - 未展示来源

### TC-009 多文档决策摘要与来源追溯

- 级别：Major
- 覆盖：`FR-005`
- 类型：Hybrid
- 前置条件：存在同一项目多份会议纪要、周报或月报
- 步骤：
  1. 选择多文档执行 summarize 或 extract_insights
  2. 查看决策、风险、待办列表
  3. 导出 Markdown
- 预期结果：
  - 每个决策、风险、待办至少带一个来源引用
  - Markdown 导出成功
  - 人工抽查 3 条洞察，均能回到源文档
  - 过程写入符合主规范的 TaskEvent
- 自动化断言：
  - 导出的 Markdown 文件存在
  - 每一类洞察至少 1 项且带引用
  - 存在 `stage_id=S5` 的结构化 TaskEvent
- 需保留证据：
  - 导出 Markdown
  - 洞察结果截图
  - TaskEvent 落盘记录
- 失败判定：
  - 洞察无来源
  - 导出失败
  - 无对应 TaskEvent 落盘

### TC-010 归档预览不执行

- 级别：Blocker
- 覆盖：`FR-006`、`FR-011`
- 类型：Hybrid
- 前置条件：
  - 至少 10 个待分类文档
  - 至少 6 个样本文件名不含项目名，且与其他项目文档混放
- 步骤：
  1. 执行分类与归档规划
  2. 查看预览清单、分类依据、目标路径、风险提示
  3. 在确认前中止
- 预期结果：
  - 自动分类结果含置信度
  - 每条建议至少给出 1 条内容依据
  - 内容证据不足时标记待确认
  - `FileOperationPlan.preview_json` 包含逐项变更、原路径/目标路径、依据摘要、风险与回滚最小信息
  - 用户取消后文件系统无任何变化
  - 归档预览阶段写入符合主规范的 TaskEvent，并带 `related_plan_id`
- 自动化断言：
  - 生成 `FileOperationPlan(status=draft)`
  - 每条分类建议包含 `confidence` 与内容依据
  - `preview_json` 可独立复核逐项预览与回滚最小信息
  - 至少 1 个中性命名样本仍能因内容被正确归类或标记待确认
  - 取消后无实际文件变动
- 需保留证据：
  - 预览清单
  - 分类依据快照
  - `FileOperationPlan.preview_json`
  - 取消后的目录快照
  - 归档预览 TaskEvent
- 失败判定：
  - 预览阶段即改动文件
  - 仅凭文件名、路径、时间给出结论
  - 无内容依据且未进入待确认
  - 无归档预览 TaskEvent 落盘

### TC-014 后台整合、M2 自动升级与冲突记忆修正

- 级别：Major
- 覆盖：`FR-008`、`FR-009`

- 步骤：
  1. 在 M2 关闭状态下触发后台整合
  2. 检查零散 M1 是否被去重、合并、更新状态，并确认未自动持久化任何 M2 偏好

  6. 执行问答触发冲突提示

  - M2 关闭时不得自动持久化偏好
  - M2 开启后，高置信偏好才可自动晋升为正式 M2

  - 冲突时明确提示

  - 低置信偏好以 `promotion_state=candidate` 持久化且未晋升为 promoted
  - 处理后的旧记忆不再参与正常召回
  - 相关操作均留下审计记录
- 需保留证据：
  - 整合前后记忆列表
  - M2 开关前后对比
  - 阈值判定快照
  - 审计日志
- 失败判定：
  - 后台整合只是堆积新条目，未去重或合并
  - M2 关闭时仍发生自动持久化
  - 低置信偏好被直接晋升为正式 M2
  - candidate 记录参与正常召回
  - 删除、禁用、修正或 supersede 后仍继续使用旧记忆

## 6. `run_acceptance.py` 参数约定

脚本入口与解释器选择以 `docs/acceptance/tasks.yaml` 中 `default_commands.acceptance` 为准。

以下只约定参数形态：

```bash
scripts/run_acceptance.py --suite smoke
scripts/run_acceptance.py --suite full
scripts/run_acceptance.py --case TC-006
scripts/run_acceptance.py --case TC-006 --llm-provider mock
scripts/run_acceptance.py --case TC-015 --local-only --llm-provider mock --embedding-provider local_model
scripts/run_acceptance.py --case TC-015 --local-only --llm-provider mock --embedding-provider local_model --assert-semantic-mode --semantic-mode normal
scripts/run_acceptance.py --case TC-015 --local-only --llm-provider mock --embedding-provider local_model --assert-semantic-mode --semantic-mode statistical_fallback
scripts/run_acceptance.py --suite perf --corpus .tmp/corpus
```

配套质量命令入口以 `docs/acceptance/tasks.yaml` 中 `default_commands.static_check`、`default_commands.coverage` 为准。

建议参数：

- `--suite smoke|full|perf|security`
- `--case TC-XXX`
- `--corpus PATH`
- `--appdata PATH`
- `--output PATH`
- `--local-only`
- `--llm-provider NAME`
- `--embedding-provider NAME`
- `--assert-semantic-mode`
- `--semantic-mode normal|statistical_fallback`
- `--report-json PATH`
- `--report-md PATH`
