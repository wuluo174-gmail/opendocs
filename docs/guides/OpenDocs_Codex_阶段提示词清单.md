# OpenDocs Codex 阶段提示词清单 v1.0

用途：你每推进一个阶段，就来这里复制当前阶段对应的提示词给 Codex。  
使用方式：**一个阶段一个线程**，先用 `Chat` 模式发“启动提示词”，确认计划后，再切到 `Agent` 发“执行提示词”。

固定规则：

- 先执行 `/local`
- 再执行 `/status`
- 打开并显式引用以下文件：
  - `docs/specs/OpenDocs_工程实施主规范_v1.0.md`
  - `docs/acceptance/tasks.yaml`
  - `docs/acceptance/acceptance_cases.md`
  - `docs/prompts/codex_operator_prompt.md`
  - `AGENTS.md`
- 每阶段前后做 Git checkpoint
- 本阶段未通过前，不得进入下一阶段

---

## S0 - 项目脚手架与基线

### 这个阶段你要先做什么
1. 在 VS Code 里打开以下文件并保持可见：
   - `docs/specs/OpenDocs_工程实施主规范_v1.0.md`
   - `docs/acceptance/tasks.yaml`
   - `docs/acceptance/acceptance_cases.md`
   - `docs/prompts/codex_operator_prompt.md`
   - `AGENTS.md`
2. 在 Codex 里确认：
   - 使用 `/local`
   - 先切到 `Chat` 模式做计划
   - 当前线程名称建议为 `S0-main`

### 复制给 Codex 的启动提示词
```text
当前只允许推进阶段 S0：项目脚手架与基线。

必须遵循以下文件，且优先级从高到低固定为：
1. docs/specs/OpenDocs_工程实施主规范_v1.0.md
2. docs/acceptance/tasks.yaml
3. docs/acceptance/acceptance_cases.md
4. docs/prompts/codex_operator_prompt.md
5. AGENTS.md

当前阶段目标：
创建可运行仓库、基础依赖、目录骨架、开发脚本、最小入口与 CI 基线。

当前阶段验收出口条件：
- 本地可安装依赖
- pytest 空跑成功
- python -m opendocs 可启动
- README 写清运行方式

当前阶段重点验收用例：
- TC-019

你的工作规则：
- 只能做 S0，不得提前进入下一阶段。
- 先复述本阶段目标、依赖、出口条件。
- 先列出将新增/修改的文件清单。
- 再开始实现。
- 完成实现后必须补齐或更新测试。
- 然后必须运行本阶段测试命令。
- 如果测试失败，只能修复当前失败，不得扩大范围。
- 如果需要偏离锁定技术栈或安全红线，必须先创建 ADR，再说明原因。
- 完成后必须输出“阶段完成报告”，包含：
  1. 新增/修改文件
  2. 关键实现说明
  3. 运行命令
  4. 测试结果
  5. 已知问题/风险
  6. 出口条件判定
  7. 下一阶段计划

本阶段测试命令：
- python scripts/bootstrap_dev.py
- python -m opendocs --help
- pytest -q

现在先只做两件事：
1. 复述你理解到的 S0 目标与出口条件；
2. 给出将新增/修改的文件清单；
不要开始下一阶段。
```

### 计划确认后，复制给 Codex 的执行提示词
```text
按刚才确认的 S0 计划开始实施。

要求：
- 只处理 S0；
- 小步修改；
- 每完成一个子任务就自检；
- 必须实际运行本阶段测试命令，而不是只说“理论上会通过”；
- 如果失败，先定位最小原因，再给最小修复；
- 在本阶段出口条件全部满足前，不得讨论 S1。

开始执行。
```

### 如果这一阶段卡住，复制这个解锁提示词
```text
不要扩大范围，不要跳阶段，不要重构整个项目。
请只处理当前 S0 的阻塞问题，并按下面格式回答：
1. 最小阻塞点是什么；
2. 为什么它阻塞了 S0；
3. 你准备做的最小修复是什么；
4. 修复后立即复跑哪些命令；
5. 如果仍失败，下一步最小排查动作是什么。
```

### 你人工要检查的 4 件事
- Codex 有没有偷偷开始做下一阶段。
- 它有没有真的运行测试命令。
- 改动的文件是否和本阶段相关。
- 报告里“出口条件判定”是否全部打勾。
