# ADR-0007: FTS5 中文分词器选型延迟到 S3/S4

## 状态

已接受（2026-03-06）

## 背景

S1 阶段的 `0001_initial.sql` 创建了 FTS5 虚拟表 `chunk_fts`：

```sql
CREATE VIRTUAL TABLE chunk_fts USING fts5(
    chunk_id UNINDEXED,
    doc_id UNINDEXED,
    text,
    content='chunks',
    content_rowid='rowid'
);
```

该表使用 SQLite FTS5 的默认分词器 `unicode61`，其行为是按 Unicode 词边界拆分。对英文和欧洲语言效果良好，但对中文（CJK）仅做逐字符切分，无法实现有意义的中文全文搜索。

主规范 `FR-003 混合智能搜索` 明确要求"支持中文搜索"，且目标用户以中文办公场景为主。

## 问题

`unicode61` 分词器下，搜索"项目进度"时无法匹配"项目进度报告"中的短语，只能匹配单个字符。这意味着当前 FTS5 索引对中文搜索几乎无用。

## 决策

在 S1 阶段不解决此问题。原因：

1. S1 的出口条件是"可初始化数据库 / 可写入查询核心实体 / CRUD 与迁移测试覆盖"，不涉及检索效果
2. FTS5 中文分词需要评估多个方案，属于 S3（索引）/ S4（检索）的职责
3. 过早锁定分词方案可能导致 S3 需要重建索引

## 候选方案（供 S3/S4 评估）

| 方案 | 优点 | 缺点 |
|------|------|------|
| ICU tokenizer (`icu`) | SQLite 官方扩展，支持中文词边界 | 需编译 ICU 扩展，增加跨平台打包复杂度 |
| `unicode61` + trigram | 不需额外依赖 | 索引膨胀，精度低 |
| 自定义 tokenizer (jieba) | 中文分词效果最好 | 需要 C 扩展或 Python 层预处理 |
| 应用层预分词 + `simple` tokenizer | 无需 SQLite 扩展 | 需在写入和查询时都做预处理 |

## 影响

- S3/S4 阶段必须在实现全文搜索前选定方案并创建新 ADR
- 当前 `chunk_fts` 表结构在更换分词器时可能需要重建（`DROP` + `CREATE`），通过新迁移脚本实现
- 向量检索（hnswlib）不受此问题影响，S4 混合检索中语义召回部分可正常工作

## 回退方案

如果 S3/S4 评估后发现所有 FTS5 中文方案都不满足打包约束，可退化为：
- 仅依赖向量检索做语义召回
- FTS5 仅服务于英文和精确匹配场景
- 需要新 ADR 记录这一降级决策
