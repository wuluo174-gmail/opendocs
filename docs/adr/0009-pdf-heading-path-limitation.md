# ADR-0009: PDF heading_path 仅基于 TOC 书签，精度为页面级

- 状态：已接受（已知限制）
- 日期：2026-03-08
- 关联阶段：S2

## 上下文

主规范 §8.3 要求 chunk 保留 `heading_path`（标题路径）。对于 txt/md/docx，可以从文本结构或样式中精确提取标题层级。但 PDF 是面向渲染的格式，文本流中没有语义化的标题标记。

## 决策

PDF 的 heading_path 使用 PDF 书签（TOC / outline）作为来源：

1. 用 PyMuPDF `doc.get_toc()` 提取书签。
2. 按页码将书签映射到对应段落。
3. 精度为**页面级**：同一页内的所有段落共享该页最后一个活跃书签的 heading_path。

### 已知限制

- 无书签的 PDF：heading_path 全部为 None。
- 同一页内多个章节：后面的章节 heading_path 会覆盖前面的。
- 扫描版 PDF（无文字层）：不在 MVP 范围内（规范 §3.5 明确排除 OCR）。

## 替代方案

- 基于字体大小/粗细推断标题：依赖 PDF 渲染细节，跨文档一致性差，实现复杂度高，MVP 阶段不值得。
- 基于正则匹配文本中的标题模式（"第X章"等）：容易误判，且不同文档风格差异大。

## 影响

- 对大多数有书签的 PDF（如 Word 导出的 PDF），heading_path 可用且合理。
- 对无书签的 PDF，不影响其他定位信息（page_no、char_start/end 仍然精确）。
- 后续可在 S4/S5 阶段通过 LLM 辅助章节识别来增强。

## 回退方案

如需更精确的标题识别，可在后续版本引入基于字体分析的增强方案，但 MVP 阶段以书签为基线已足够。
