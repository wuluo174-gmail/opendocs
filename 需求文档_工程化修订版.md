## 1. 执行摘要

### 1.1 产品目标

OpenDocs 要解决的是：个人电脑里存在大量凌乱、重复、命名不统一、跨项目混放的工作文档，用户能“看见文件”，却无法“快速找对文件、用对内容、正确归档、在证据基础上生成新文档”。

本项目的 MVP 必须让用户完成以下高频闭环：

1. **找得到**：用户可以用自然语言找到目标文档与证据片段。
2. **问得准**：用户可以对跨文档问题获得有引用的回答；证据不足时系统明确拒答。
3. **理得清**：系统可以给出分类、标签、归档建议，并在确认后安全执行。
4. **写得出**：系统可以基于历史文档生成有引用的纪要、周报、月报草稿。
5. **可复核**：所有重要操作都有审计、引用、预览与回滚。

### 1.2 产品边界

本项目**不**做以下事情：

- 不做通用电脑控制器
- 不做浏览器自动化平台
- 不做组织级协同办公平台
- 不做扫描版 OCR 识别
- 不做默认删除文件
- 不做复杂图数据库驱动的企业知识中台

### 1.3 与 OpenClaw 的关系

OpenDocs 仅借鉴 OpenClaw 的“Agent 编排思想、计划预览后执行、工具分级授权”等工程观念，但不复制其广泛场景覆盖范围。OpenClaw 面向广义个人 AI 助手场景；OpenDocs 只聚焦**本地文本类文档管理**这一窄而深的垂直任务域。

### 1.4 成功定义

当以下条件全部满足时，可视为 MVP 验收通过：

- 在 1 万文档规模下完成稳定索引与增量更新
- 搜索、问答、摘要、分类、归档、生成、记忆、审计全链路可运行
- 事实性回答默认附证据引用
- 所有写操作均有预览、确认、审计和最近一次批量回滚
- 纯本地模式下无外网请求
- 跨 Windows / macOS / Linux 核心用例通过
- 自动化验收脚本全部通过

## 2. 多学科设计依据与工程化转译

### 2.1 系统工程学转译

本项目采用“需求 -> 设计 -> 实现 -> 验证 -> 验收”的闭环治理。每项能力都必须具备：

- 明确的输入输出
- 可验证的验收标准
- 可追踪的数据对象
- 可定位的故障边界
- 明确的阶段出口条件

工程上要求建立**需求追踪矩阵、阶段门禁、自动化测试、验收脚本**，避免只有 PRD 没有施工规约。[REF-01][REF-02]

### 2.2 档案管理学转译

OpenDocs 管理的是“工作文档”，其很多内容具有“记录（record）”特征，因此系统设计必须遵守以下原则：

- **来源可追溯**：任何摘要、回答、标签、生成内容都要能回溯到原文
- **元数据完整**：文档必须具备来源路径、时间、类型、哈希、索引时间等元数据
- **原件保护优先**：默认不破坏原目录结构，优先做虚拟归档
- **可复核**：自动分类、摘要、洞察、生成都必须能被人复查
- **职责明确**：系统执行写操作时必须记录执行对象、时间、计划与结果

这使 OpenDocs 更接近“带 AI 能力的轻量记录管理系统”，而不是普通聊天检索器。[REF-03][REF-04]

### 2.3 应用开发科学转译

为降低 AI 施工复杂度与跨平台风险，本项目采用**单语言优先、模块边界清晰、数据落地简单、默认本地化**的工程策略：

- 优先选用 Python 单栈完成 MVP
- 用 SQLite 承担元数据、审计、任务记录和 FTS 检索
- 用 HNSW 向量索引承担语义召回
- 用桌面应用承载“可视化计划预览、证据检查、确认执行”
- 所有高风险写操作强制走“预览 -> 确认 -> 执行 -> 可回滚”

### 2.4 脑科学、神经科学与认知工程转译

OpenDocs 的“记忆设计”和“交互设计”不能照搬聊天机器人套路，而应遵循以下人因规律：

- **Recognition over recall**：界面应让用户“看见可选动作、看见证据、看见计划”，而不是要求用户记住路径、命令和上下文。[REF-06]
- **Memory consolidation**：只有被反复使用、被明确确认、与任务完成强相关的信息，才值得从临时上下文提升为持久任务记忆或偏好记忆。[REF-07]
- **Forgetting / decay**：陈旧的任务状态与偏好必须自动衰减；否则会造成“记忆污染”。
- **Interference control**：事实知识与交互记忆必须分离，防止过期记忆干扰基于文档证据的事实回答。

### 2.5 生成式 AI 风险治理转译

本项目把“治理、映射、测量、管理”视作生成式 AI 系统的基本控制面，因此必须在产品内建：

- 模型路由策略
- 外发最小化策略
- 引用校验机制
- 风险审计与日志
- 失败拒答机制
- 隐私模式与提供商开关

这使系统从一开始就具备“可信 AI 应用”的最小治理框架。[REF-05]

## 3. 产品定义与范围

### 3.1 产品定义

OpenDocs 是一个安装在个人电脑上的桌面 AI 组件。它负责：

- 监控和扫描用户指定的文档目录
- 解析文本类文档并建立混合索引
- 提供可引用搜索、问答、摘要、洞察
- 支持虚拟归档与受控物理归档
- 管理任务记忆与用户偏好记忆
- 输出审计日志、执行计划与回滚记录

### 3.2 目标用户

1. 办公人员：项目周报、月报、会议纪要、招投标、制度材料
2. 研究人员：论文、实验记录、研究日志、阶段报告
3. 自由职业者与顾问：客户材料、合同、交付件、报价与复盘文档

### 3.3 核心 Jobs To Be Done

- “帮我找到某项目上个月的进度报告，并给我引用依据”
- “汇总最近三个月会议纪要里的决策、风险和待办”
- “请把桌面上和 A 项目有关的文档整理出来，先给我看方案，再执行归档”
- “基于最近四周的周报，生成一份本月月报草稿，并附引用”

### 3.4 MVP In Scope

- 文档源配置、递归扫描、排除规则
- 文本类文档解析：`.txt`、`.md`、`.docx`、文字层 `.pdf`
- 全量索引、增量索引、重建索引
- 混合搜索：关键词 + 向量检索 + 过滤器
- RAG 问答与证据引用
- 单文档摘要、多文档汇总、主题聚合
- 虚拟归档、受控物理归档（move / rename）
- 模板生成与自由生成
- M0 / M1 / M2 记忆体系
- 模型路由、隐私模式、审计、回滚

### 3.5 MVP Out of Scope

- OCR
- 图片理解
- 音视频转写
- 网页抓取
- 多人协同审批
- 复杂图数据库
- 默认删除文件
- 自动发送邮件或跨应用自动化

### 3.6 运行假设

- 单机单用户
- 默认文档规模 <= 10,000 个文件
- 文本总量建议 <= 20 GB
- 基线机器配置：
  - CPU：4 核及以上
  - 内存：16 GB 及以上
  - 剩余磁盘：20 GB 及以上
- 用户具备至少一个可用模型提供方：
  - 本地：Ollama
  - 云端：OpenAI / Anthropic / 通义 / 智谱中的任意一种

## 4. 锁定技术决策（不得随意替换）

> 本节是“AI 施工去歧义”的关键。ChatGPT-Codex / 类 Codex 代理必须把它视为默认实现基线。  
> 若需替换，必须先提交 ADR（Architecture Decision Record），并说明替换原因、兼容影响、测试影响与回退方案。

### 4.1 总体技术栈

- 语言：Python 3.11
- 桌面界面：PySide6
- 数据校验：Pydantic v2
- 数据库：SQLite 3（含 FTS5）
- ORM / 数据访问：SQLAlchemy 2.x
- 向量索引：hnswlib
- 文件监听：watchdog
- 文档解析：
  - `.txt`：内置读取
  - `.md`：Markdown 文本解析
  - `.docx`：python-docx
  - `.pdf`：PyMuPDF 优先，失败时降级 pypdf（仅文字层）
- 模板：Jinja2
- 测试：pytest、pytest-qt、pytest-cov
- 打包：PyInstaller
- 配置：TOML + 环境变量
- 日志：标准 logging + 结构化 JSON 日志
- 密钥存储：keyring（系统钥匙串 / 凭据管理器）

### 4.2 架构风格

MVP 采用**模块化单体（modular monolith）**，而非微服务：

- 减少 AI 施工复杂度
- 降低本地部署成本
- 降低跨平台打包风险
- 保留未来服务化的演进空间

### 4.3 检索策略

默认采用**混合检索**，禁止只做纯向量检索或只做关键词检索：

- 词法召回：SQLite FTS5
- 语义召回：hnswlib
- 重排序：基于分数融合 + 元数据过滤 + 新鲜度加权
- 输出：必须携带可引用片段

### 4.4 记忆策略

- 记忆与知识分库、分表、分流程
- M0 会话记忆不持久化
- M1 任务记忆持久化，默认 TTL 30 天
- M2 用户偏好记忆默认关闭，显式开启才持久化
- 任何记忆均不得作为事实证据替代文档证据

### 4.5 写操作策略

- 默认只读优先
- 物理写操作只允许：move、rename、create
- delete 默认禁用
- 所有写操作必须预览、确认、审计、可回滚

### 4.6 知识图谱策略

基于用户长期偏好，本项目**不做“简单切块 + 简单 RAG”**。  
但为了控制 MVP 复杂度，MVP 只实现“轻量关系层（Relation Edge）”，而不引入完整图数据库。

具体做法：

- 保留 `relation_edge` 数据表，用于存储跨文档实体、主题、引用、任务关联
- 用它增强聚类、推荐和跨文档摘要
- 完整知识图谱、图查询语言、复杂推理引擎放入后续版本

## 5. 系统目标、KPI 与验收结果

### 5.1 业务目标

- 帮助个人用户管理混乱文本工作资料
- 将“找文件”“读文件”“写新文档”“归档文件”合并到同一工具中
- 用可证据化回答替代传统“拍脑袋式文档问答”

### 5.2 关键成功指标（Product KPI）

- 搜索成功率：在黄金测试集上，目标文档出现在 Top10 的比例 >= 90%
- 回答引用覆盖率：事实型回答中带引用的比例 >= 95%
- 证据不足拒答正确率 >= 90%
- 分类建议可接受率 >= 80%
- 批量归档回滚成功率 >= 99.9%
- 纯本地模式外发请求数 = 0

### 5.3 用户体验指标（UX KPI）

- 首次索引流程可在 3 步内完成
- 用户在主界面 10 秒内可以找到“搜索 / 问答 / 归档建议 / 审计”入口
- 归档前预览中必须一眼看清：
  - 哪些文件会变动
  - 将移动到哪里
  - 为什么这么分类
- 任何答案的引用面板必须可一键展开

## 6. 系统上下文与总体架构

### 6.1 外部上下文

系统外部对象包括：

- 本地文件系统
- 本地模型运行环境（Ollama）
- 云端模型提供方 API
- 用户桌面操作系统
- 系统凭据管理器（keyring）

### 6.2 内部组件

OpenDocs 由以下核心组件组成：

1. **Desktop UI**
   - PySide6 实现
   - 负责配置、检索、问答、预览、确认、审计、回滚界面

2. **Application Service Layer**
   - 统一业务编排层
   - 不直接包含 UI 逻辑和底层存储细节

3. **Parser Registry**
   - 负责识别文件类型并调用对应解析器
   - 输出标准化纯文本与结构化元数据

4. **Indexer**
   - 负责全量 / 增量索引
   - 负责 chunk 生成、哈希、FTS 入库、向量入库

5. **Retrieval Engine**
   - 负责搜索、过滤、分数融合、重排序与证据构造

6. **QA / Insight Engine**
   - 负责问答、摘要、决策提取、风险提取、待办提取

7. **Classification & Archive Planner**
   - 负责标签、分类、归档建议、路径规划、冲突检测

8. **Generation Engine**
   - 负责模板生成、自由生成、引用嵌入、保存预览

9. **Memory Service**
   - 负责 M0/M1/M2 记忆读写与 TTL 处理
   - 负责记忆污染控制与冲突提示

10. **Provider Router**
    - 负责本地 / 云端模型路由
    - 负责最小必要片段外发

11. **Audit & Rollback Service**
    - 负责操作日志、任务日志、最近一次批量回滚

12. **Config & Security Service**
    - 负责配置读取、密钥管理、敏感度策略、隐私模式

### 6.3 总体流程

#### 流程 A：扫描与索引

1. 用户配置根目录与排除规则
2. 扫描器枚举文件
3. 解析器抽取文本和元数据
4. Chunker 切片并附带定位信息
5. 元数据写入 SQLite
6. 词法索引写入 FTS5
7. 语义向量写入 HNSW
8. 记录 audit 与 index_run

#### 流程 B：搜索与问答

1. 用户输入自然语言问题
2. 系统解析过滤条件与任务类型
3. 检索引擎执行 FTS + 向量召回
4. 构造候选证据集合
5. 检查记忆是否相关，但不允许记忆替代证据
6. 交给模型生成答案草稿
7. 引用校验器检查答案中的陈述是否有证据支撑
8. 返回答案、引用、冲突说明或证据不足提示

#### 流程 C：分类与归档

1. 用户选择目录或搜索结果集
2. 分类器输出标签、分类、目标路径、置信度
3. 系统生成归档计划
4. UI 展示“当前路径 -> 目标路径”的变更列表
5. 用户确认后执行 move / rename
6. 写审计日志
7. 保留最近一次批量回滚记录

#### 流程 D：生成与落盘

1. 用户选择模板或自由生成
2. 系统检索相关文档证据
3. 模型生成草稿
4. 系统插入引用标记
5. 用户编辑
6. 用户确认保存到输出目录
7. 写审计日志

### 8.2 数据不变量

以下不变量是强约束：

1. 所有事实性回答必须关联至少 1 个 `chunk_id`
2. 所有 `KnowledgeItem` 必须能回溯到 `Document + Chunk`
3. `MemoryItem` 不得直接用于生成事实引用
4. 所有 `move / rename / create` 必须先生成 `FileOperationPlan`
5. 所有执行过的写操作必须有对应 `AuditLog`
6. 最近一次批量写操作必须可回滚
7. 删除操作默认无实现入口

### 8.3 切片规则（Chunking）

默认切片规则如下：

- 优先按标题和段落边界切片
- 当段落过长时按 token 预算切片
- 默认目标：
  - 中文：600 到 1200 字符
  - 英文：350 到 700 tokens
- 重叠比例：10% 到 15%
- 片段必须保留：
  - 文档 ID
  - 文档内顺序
  - 页码或段落号
  - 字符偏移
  - 标题路径

### 8.4 引用格式规范

内部引用对象采用统一结构：

```json
{
  "doc_id": "uuid",
  "chunk_id": "uuid",
  "path": "Projects/A/weekly_report_2026-02.md",
  "page_no": 3,
  "paragraph_range": "5-6",
  "char_range": "1280-1672",
  "quote_preview": "……"
}
```

UI 展示格式：

- `来源：Projects/A/weekly_report_2026-02.md，第 3 页，第 5-6 段`
- 若无页码：`来源：Projects/A/weekly_report_2026-02.md，第 12-13 段`

## 10. 详细功能需求（可直接施工）

### FR-001 文档源配置与扫描

- 优先级：P0
- 目标：允许用户配置 1 个或多个根目录，建立可重复扫描的来源集合
- 依赖：配置服务、扫描服务、解析器注册表

#### 输入

- 根目录路径
- 排除规则（目录名、后缀、大小上限、隐藏文件策略）
- 是否递归
- 是否立即扫描

#### 主流程

1. 用户新增根目录
2. 系统验证路径存在且可读
3. 系统保存配置
4. 用户点击“开始扫描”
5. 系统递归遍历并过滤
6. 输出纳入 / 排除 / 失败统计

#### 异常流程

- 路径不存在 -> 阻止保存并提示
- 无权限读取 -> 标记为失败，不中断整体任务
- 文件格式不支持 -> 记入排除清单

#### 验收标准

- 能保存并加载文档源配置
- 可展示纳入 / 排除 / 失败数量
- 单文件失败不应导致全任务中断
- 扫描完成后生成 `scan_run` 记录与审计记录

#### 自动化测试

- 正常路径扫描
- 不存在路径
- 无权限路径
- 排除规则命中
- 大目录扫描稳定性

- 优先级：P0

- 新增、修改、删除能反映到索引
- 索引失败支持重试
- 删除的文件在检索结果中不再返回
- 重建索引不会产生脏重复数据

#### 自动化测试

- 新文件增量入库
- 已有文件修改重建 chunk
- 文件删除索引同步
- 故障恢复与重试
- 重建幂等性

### FR-003 混合智能搜索

- 优先级：P0
- 目标：以自然语言和过滤条件查找文档与证据片段

#### 检索要求

- 必须实现 FTS5 + 向量召回融合
- 支持过滤：
  - 根目录
  - 分类
  - 标签
  - 文件类型
  - 时间范围
  - 敏感等级
- 返回结果字段：
  - 标题
  - 路径
  - 片段摘要
  - 时间
  - 相关性分数
  - 证据定位信息

#### 默认打分建议

```text
hybrid_score = 0.55 * lexical_score
             + 0.35 * dense_score
             + 0.10 * freshness_boost
```

若用户指定目录、项目、标签，则可额外加入过滤后 boost。

#### 验收标准

- TopN 结果可打开源文件与所在目录
- 每个结果至少包含 1 个可引用片段
- 支持中文搜索
- 对同义表达具备一定鲁棒性

#### 自动化测试

- 黄金查询集 Top10 命中率
- 过滤组合查询
- 中文 / 英文 / 混合语料
- 无结果场景
- 相关性排序基本正确性

### FR-008 记忆体系

- 优先级：P1
- 目标：实现 M0 / M1 / M2 三层记忆

#### 定义

- M0 会话记忆：仅当前会话有效，不落盘

- 明确确认一次或高频出现三次以上，才允许从 M0 升为 M1 / M2

- 长期不使用的偏好记忆降低召回优先级

#### 验收标准

- 记忆开关实时生效
- 清理后不可再被召回
- 记忆读取与写入具备审计记录

#### 自动化测试

- M0 生命周期

- M2 显式开关

- 审计记录存在

## 12. AI 编排、提示词协议与回答契约

### 12.1 Agent 模式定义

OpenDocs 内置的文档 Agent 不是自由发挥的聊天机器人，而是受控编排器。其工作顺序固定为：

1. 识别任务意图
2. 判断是否需要读取记忆
3. 检索文档证据
4. 检查冲突与证据充分性
5. 生成答案或计划
6. 若涉及写操作，则进入预览与确认
7. 写审计日志

### 12.2 系统级不可违背规则

1. 文档证据优先于记忆
2. 事实性结论必须带引用
3. 证据不足时必须拒答
4. 不允许未确认就执行物理写操作
5. 不允许在纯本地模式外发数据
6. 不允许将完整长文档默认发到云端
7. 不允许隐藏冲突和不确定性

### 12.3 回答模板

#### 模板 A：事实问答

```text
结论：
- ...

依据：
1. 来源：...
2. 来源：...

不确定性：
- ...
```

#### 模板 B：证据不足

```text
当前证据不足以可靠回答该问题。

已检查来源：
1. ...
2. ...

建议下一步：
- 扩大检索范围
- 指定时间范围 / 项目 / 目录
```

#### 模板 C：冲突证据

```text
发现冲突信息，暂不输出单一结论。

冲突来源 A：
- ...

冲突来源 B：
- ...

建议：
- 由用户确认哪个版本为最新
- 或查看时间更晚、级别更高的记录
```

#### 模板 D：归档预览

```text
建议操作：move / rename

预计影响文件数：N
分类依据：
- ...

变更清单：
- 当前路径 -> 目标路径
- ...

风险提示：
- ...
```

### 12.4 引用校验器要求

在模型生成候选答案后，必须执行二次校验：

- 检查每个关键事实是否被至少一个证据片段支撑
- 检查引用对象是否真实存在
- 检查引用定位是否可打开
- 检查引用是否与结论相反
- 若不通过，则降级为：
  - 局部删改
  - 输出冲突说明
  - 证据不足拒答

### 12.5 记忆调用策略

- 查询型任务：先检索证据，再补充记忆
- 执行型任务：可先读取任务记忆恢复上次中断点
- 偏好型任务：只有当 M2 开启且该偏好被显式确认时才调用

### 12.6 模型路由建议

- 文件解析、检索、过滤、排序：本地逻辑，不走 LLM
- 摘要、问答、洞察、生成：可走 LLM
- 路由优先级：
  1. Local-Only：Ollama
  2. Hybrid：本地模型优先，长生成或高复杂度任务可切云端
  3. Cloud-Assisted：按提供商策略

## 14. 安全、隐私与可信性要求

### 14.1 权限与授权分级

| 级别 | 操作 | 执行方式 |
| --- | --- | --- |
| L1 自动执行 | search, read, list, summarize, classify, tag, index, memory_search | 自动执行 |
| L2 需确认 | move, rename, create, memory_write(M2), save_output | 计划预览后确认 |
| L3 禁止 / 双重确认 | delete, purge_memory_all, external_export_bulk | 默认禁用，开启后双重确认 |

### 14.2 隐私红线

- 纯本地模式不得发起外网请求
- 云端模式不得发送整个目录的原文
- 外发必须最小化
- 密钥不得写入日志
- 审计导出默认不包含原文正文，只包含摘要与定位信息

### 14.3 文件系统安全

- 所有物理写操作都记录前后路径
- 目标路径冲突时默认停止，不自动覆盖
- 批量操作前进行路径合法性检查
- 只允许在用户已配置根目录内执行归档；默认不跨盘写入

### 14.4 可信性机制

- 引用校验
- 冲突检测
- 审计留痕
- 回滚能力
- 提供商可切换
- 网络模式可视化

## 20. 风险清单与缓解策略

| 风险 | 描述 | 缓解策略 |
| --- | --- | --- |
| 记忆污染 | 过时偏好或任务状态影响回答 | TTL、确认升级、冲突检测、手动清理 |
| 幻觉回答 | 未基于证据输出事实 | 引用校验器、证据不足拒答 |
| 文件误操作 | 批量 move / rename 造成混乱 | 计划预览、用户确认、最近一次批量回滚 |
| 索引漂移 | 文件变化后索引未更新 | 监听、定时校验、重建索引 |
| 隐私泄露 | 云端外发过多内容 | 最小必要片段外发、可视审计 |
| 打包复杂度 | 桌面应用跨平台打包失败 | 单语言栈、PyInstaller、阶段化打包 |
| 检索效果差 | 只做简单 RAG 导致相关性低 | 混合检索、轻量关系层、黄金问题集校准 |
| UI 复杂度失控 | 一次性堆太多功能 | 严格按阶段做 MVP 界面 |
| 提供商不稳定 | 云端接口变化导致失败 | Provider Adapter、Mock 测试、本地模式兜底 |

## 22. 给 ChatGPT-Codex / 类 Codex 的施工协议

> 本节可直接复制给 AI 编码代理作为项目执行契约。

### 22.1 角色定义

你是本项目的主施工代理，任务不是“讨论想法”，而是**按本规范从零实现 OpenDocs，直到通过自动化验收与人工验收**。

### 22.2 施工总规则

1. 把本文件视为最高优先级规范
2. 严格按“阶段 0 -> 阶段 11”顺序推进
3. 不得跳过测试直接进入下阶段
4. 不得把未确认的设计变动悄悄写进代码
5. 若需改变锁定技术栈，先产出 ADR 再实施
6. 遇到本文档已明确的事项，不要重复询问
7. 遇到本文档未明确但可合理推断的事项，按“安全优先、证据优先、本地优先、简单优先”原则自行决策并记录
8. 高风险功能必须先做最小可行版本，再扩展
9. 不得引入与 MVP 无关的大型框架
10. 任何会影响文件安全、隐私或审计闭环的设计，必须优先落实

### 22.3 每阶段输出契约

每完成一个阶段，必须输出：

- 本阶段新增 / 修改文件列表
- 关键实现说明
- 运行命令
- 测试结果
- 已知问题
- 是否满足出口条件
- 下一阶段计划

### 22.4 遇到阻塞时的处理规则

若遇到阻塞：

1. 先判断能否通过降级方案继续推进
2. 能降级则继续施工，并记录 trade-off
3. 不能降级时，产出：
   - 阻塞原因
   - 已尝试方案
   - 推荐最小改动
   - 影响范围
4. 不要因为小歧义停工等待

### 22.5 代码质量规则

- 优先清晰可维护，而不是炫技
- 核心域逻辑必须有类型标注
- 函数职责单一
- 错误要可定位
- 禁止把业务规则散落在 UI
- 核心服务必须可单测
- 必须保留结构化日志

### 22.6 安全红线

- 不得默认实现 delete
- 不得在无确认情况下执行 move / rename / create
- 不得在 Local-Only 模式中发起外网请求
- 不得将 API key 写入配置文件或日志
- 不得让记忆覆盖文档证据

### 22.7 结束判定

只有当以下全部满足时，施工才算完成：

- 阶段 0 ~ 11 全部完成
- 自动化验收通过
- README、配置示例、用户手册完整
- 三平台至少形成可运行发布目录
- 已知 blocker 数量为 0
